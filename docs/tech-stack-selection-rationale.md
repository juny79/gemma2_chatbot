## 1. 선택 기준 개요

### 1.1 공통 평가 기준 (우선순위 순)

본 프로젝트의 모든 기술 선택은 아래 5가지 기준을 우선순위 순으로 적용하여 결정하였다.

| 우선순위 | 기준 | 설명 |
|---------|------|------|
| 1순위 | **하드웨어 적합성** | RTX 3070 8GB VRAM 내에서 안정적으로 동작 가능한가 |
| 2순위 | **메모리 효율성** | VRAM 및 RAM 사용량이 최적화되어 있는가 |
| 3순위 | **기술 성숙도** | 프로덕션 환경에서 검증된 안정적인 기술인가 |
| 4순위 | **개발 생산성** | 개발 속도, 문서 품질, 커뮤니티 지원이 충분한가 |
| 5순위 | **확장성** | 향후 기능 추가 및 규모 확장 시 대응 가능한가 |

### 1.2 제약 조건

```
하드웨어 제약:
  - VRAM 상한: 8GB (실사용 가능: ~7.0GB)
  - RAM 상한: 16GB (OS 포함)
  - GPU: NVIDIA RTX 3070 (CUDA 11.8+)
  - 네트워크: 고정 IP 없음 (Cloudflare Tunnel 필요)

소프트웨어 제약:
  - OS: Windows 10/11 또는 Ubuntu 22.04
  - 외부 서버 없음 (완전 로컬 운영)
  - 오픈소스 우선 (라이선스 비용 최소화)
```

## 2. AI 추론 계층 선택 근거

### 2.1 모델: `solidrust/gemma-2-9b-it-AWQ`

#### 선택 이유

```
일반 Gemma-2-9b-it (FP16): 약 18GB VRAM 필요 → RTX 3070 8GB 불가
Gemma-2-9b-it-AWQ (4-bit): 약 5.5GB VRAM 필요 → RTX 3070 8GB 가능 ✅
```

| 평가 항목 | 점수 | 근거 |
|---------|------|------|
| 하드웨어 적합성 | ★★★★★ | 5.5GB로 8GB 한계 내 수용 가능 |
| 성능 | ★★★★☆ | 4-bit 양자화 대비 성능 저하 최소 (AWQ 특성) |
| 언어 능력 | ★★★★★ | 한국어 포함 다국어 지원 우수 |
| 라이선스 | ★★★★★ | 오픈소스, 상업적 이용 가능 |

**AWQ(Activation-aware Weight Quantization) 선택 이유**:
- 단순 4-bit INT 양자화(GPTQ 등) 대비 **활성화 분포를 고려한 가중치 양자화**로 성능 저하 최소화
- 추론 속도: FP16 대비 동등하거나 빠름
- 동일 메모리에서 GPTQ 대비 품질 우수

---

### 2.2 추론 라이브러리: `AutoAWQ` + `transformers`

#### 선택 이유

| 평가 항목 | AutoAWQ ⭐ | TGI (대안) | Ollama (대안) |
|---------|-----------|-----------|-------------|
| AWQ 네이티브 지원 | ✅ | ✅ | ❌ (GGUF만) |
| Windows 호환성 | ✅ | ❌ (Docker 필요) | ✅ |
| 설정 복잡도 | 낮음 | 높음 | 매우 낮음 |
| 프로덕션 적합성 | 중간 | 높음 | 낮음 |
| HuggingFace 직접 연동 | ✅ | ✅ | ❌ |
| **종합 추천** | **1순위** | 동시 10+ 요청 시 | 데모 전용 |

**결론**: Windows/Linux 모두 지원하면서 AWQ를 네이티브로 다루는 유일한 경량 옵션

---

### 2.3 딥러닝 프레임워크: `PyTorch ≥2.1 (CUDA 12.x)`

#### 선택 이유

- **RTX 3070 최적화**: CUDA 12.x + PyTorch 2.1은 Ampere 아키텍처(RTX 30 시리즈) 최적화 컴파일러 포함
- **torch.compile()**: PyTorch 2.0+ 에서 추가된 컴파일러로 추론 속도 최대 30% 향상
- **업계 표준**: HuggingFace 전 생태계가 PyTorch 기반 → 호환성 문제 없음
- **대안 TensorFlow**: 동일 기능이나 HuggingFace 생태계와 통합성 낮음

## 3. 백엔드 API 계층 선택 근거

### 3.1 웹 프레임워크: `FastAPI ≥0.110`

#### 선택 이유

| 평가 항목 | FastAPI ⭐ | Django | Flask | Starlette |
|---------|----------|--------|-------|----------|
| 비동기 I/O 지원 | 네이티브 | 제한적 | 제한적 | 네이티브 |
| 타입 안전성 | Pydantic 내장 | 별도 구성 | 별도 구성 | 없음 |
| 자동 API 문서화 | ✅ (Swagger/ReDoc) | 별도 필요 | 별도 필요 | ❌ |
| WebSocket 지원 | ✅ 내장 | 채널 필요 | 별도 라이브러리 | ✅ 내장 |
| SSE 스트리밍 | `StreamingResponse` 기본 지원 | 제한적 | 제한적 | ✅ |
| RAM 사용량 | ~0.3-0.5 GB | ~0.8-1 GB | ~0.2-0.3 GB | ~0.2 GB |
| **선택 이유** | **AI 스트리밍 + 타입 안전성 최적** | 과중량 | 비동기 약함 | 기능 부족 |

**핵심 선택 이유**: Gemma-2 토큰 스트리밍을 SSE(Server-Sent Events)로 클라이언트에 실시간 전달하는 구조에서 `StreamingResponse`가 기본 내장된 FastAPI가 가장 적합하다.

---

### 3.2 ASGI 서버: `Uvicorn ≥0.29` (단일 워커)

#### 선택 이유

- **단일 워커 권장 이유**: AI 추론 서버가 GPU를 독점적으로 사용 → 멀티 워커 구성 시 VRAM 충돌 발생
- **Gunicorn 미선택 이유**: GPU 사용 환경에서 멀티프로세스 모델은 VRAM을 프로세스 수만큼 배율로 요구
- **hypercorn 미선택 이유**: Uvicorn 대비 설정 복잡도 높고, 이점 미미

```bash
# RTX 3070 환경에서 권장 실행 방식
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 1
#                                                       ↑ 단일 워커 필수
```

---

### 3.3 인증: `python-jose` + `passlib`

#### 선택 이유

- **JWT 선택 이유**: 세션 기반 인증은 Redis 의존성 추가 → JWT로 stateless 인증 유지
- **python-jose**: FastAPI 공식 문서 표준 JWT 구현체
- **passlib**: bcrypt 해싱 지원, 보안 취약점 이력 없음

## 4. 데이터 계층 선택 근거

### 4.1 관계형 DB: `PostgreSQL 15`

#### 선택 이유

| 평가 항목 | PostgreSQL ⭐ | MySQL 8 | SQLite |
|---------|-------------|---------|--------|
| JSON/JSONB 지원 | 우수 (인덱싱 가능) | 보통 | 없음 |
| 동시 접속 처리 | 우수 (MVCC) | 보통 | 약함 |

| RAM 사용 | ~1-1.5 GB | ~0.8-1 GB | ~0 GB |
| **선택 이유** | **JSON 채팅 데이터 + 동시성 최적** | 대안 가능 | 단일 사용자 전용 |

**채팅 히스토리 저장 구조 관점**: 대화 메시지는 구조적 JSON(role, content, timestamp 등)이므로 JSONB 타입을 지원하는 PostgreSQL이 쿼리 효율 면에서 최적이다.

## 7. 인프라 및 DevOps 선택 근거

### 7.1 외부 노출: `Cloudflare Tunnel`

#### 선택 이유

| 평가 항목 | Cloudflare Tunnel ⭐ | ngrok | Frp | SSH 터널 |
|---------|-------------------|-------|-----|---------|
| 비용 | **무료** | 무료/유료 (제한) | 무료 (서버 필요) | 무료 |
| 고정 URL | ✅ (사용자 지정 도메인) | 유료 플랜 | ✅ | ❌ |
| HTTPS 자동화 | ✅ (자동 TLS) | ✅ | 별도 설정 | ❌ |
| DDoS 방어 | ✅ (Cloudflare CDN) | ❌ | ❌ | ❌ |
| 안정성 | 최고 | 중간 | 중간 | 낮음 |
| **선택 이유** | **무료 + 고정 URL + TLS + DDoS 방어** | 개발/테스트용 | 자체 서버 필요 | 임시 전용 |

**고정 IP 없는 환경에서 유일한 무료 프로덕션급 솔루션**.

---

### 7.2 컨테이너: `Docker + Docker Compose`

#### 선택 이유

- **재현성**: "내 PC에서는 되는데" 문제 제거 → Docker로 환경 고정
- **서비스 격리**: PostgreSQL, Redis, ChromaDB 등 각 서비스를 독립 컨테이너로 분리 → 포트 충돌, 의존성 충돌 방지
- **GPU 지원**: `deploy.resources.reservations.devices` 설정으로 컨테이너에 RTX 3070 직접 할당
- **Kubernetes 미선택 이유**: 단일 PC 환경에서 Kubernetes는 오버엔지니어링, 메모리 ~4-5GB 추가 사용

---

### 7.3 CI/CD: `GitHub Actions + Self-hosted Runner`

#### 선택 이유

| 평가 항목 | GitHub Actions + Self-hosted ⭐ | GitHub Actions (Cloud) | Jenkins |
|---------|-------------------------------|----------------------|---------|
| GPU 접근 | ✅ (로컬 Runner가 RTX 3070 직접 사용) | ❌ (GPU 없음) | ✅ (로컬 설치 시) |
| AI 모델 테스트 | ✅ (실제 추론 테스트 가능) | ❌ | ✅ |
| 비용 | **무료** | 무료 (제한) | 무료 (서버 필요) |
| GitHub 통합 | 최고 | 최고 | 플러그인 필요 |
| **선택 이유** | **로컬 GPU로 실제 AI 테스트 + GitHub 통합** | GPU 없음 | 설정 복잡 |

**핵심**: AI 추론 서버의 통합 테스트를 위해 **실제 RTX 3070에서 모델 추론 테스트**가 필요 → Self-hosted Runner가 필수.

---

### 7.4 AI 에이전트 프레임워크: `LangChain`

#### 선택 이유

- **코드 리뷰 에이전트 구현**: PR diff를 Gemma-2에 전달하고 리뷰 코멘트를 구조화하는 파이프라인 구성에 최적
- **RAG 파이프라인**: ChromaDB + LangChain 조합으로 지식 기반 검색 구현 간소화
- **대안 LlamaIndex**: 문서 인덱싱에 특화, 에이전트 기능은 LangChain이 더 성숙
- **직접 구현 미선택 이유**: 토큰 분할, 컨텍스트 관리, 도구 호출 등을 직접 구현 시 개발 기간 과다 소요

### 8.2 기술 선택의 핵심 원칙

```
1. VRAM 우선 원칙
   → 모든 기술 선택의 출발점은 "RTX 3070 8GB 내에서 동작하는가?"

2. 경량화 원칙
   → 동일 목적이면 더 가벼운 기술 선택 (RAM/VRAM 절약 우선)

3. 단일 GPU 보호 원칙
   → 멀티 워커, 멀티 프로세스 구성 지양 → GPU 독점 보장

4. 확장성 내재 원칙
   → Redis Pub/Sub, LangChain, n8n 등 향후 확장 포인트를 초기 설계에 반영

5. 오픈소스 우선 원칙
   → 비용 발생 없이 완전 로컬 운영 가능한 기술 선택
```

### 8.3 기술 교체 시나리오
